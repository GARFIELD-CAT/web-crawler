# Распределённый веб-краулер (учебный проект)

Распределённая система для параллельного сбора данных с веб-сайтов на C# и .NET 10.
Проект демонстрирует современные подходы к параллельному и распределённому
программированию: `async/await`, **TPL Dataflow**, **PLINQ**, потокобезопасные
коллекции, атомарные операции `Interlocked`, а также **мастер-рабочую архитектуру**
с обменом по TCP, «пульсом» (heartbeat), переназначением задач при отказе узлов,
повторными попытками (ретраями) при временных ошибках и выгрузкой результатов в CSV.

> Стартовый сайт по умолчанию — `https://books.toscrape.com` — учебный
> «песочница»-сайт, специально созданный для практики краулинга. Его можно
> безопасно обходить и менять на другой через параметр `--seed`.

---

## Что умеет система

- **Параллельная загрузка** веб-страниц (асинхронные HTTP-запросы).
- **Извлечение данных** из HTML: заголовок, слова, ссылки.
- **Обход ссылок** с ограничением глубины и только в пределах одного домена.
- **Индексация и поиск** по собранным данным (инвертированный индекс + поиск через PLINQ).
- **Распределение задач** между несколькими рабочими узлами (воркерами).
- **Мониторинг** состояния системы в реальном времени и итоговая статистика.
- **Отказоустойчивость**: если воркер «падает», его задачи получают другие.
- **Ретраи**: временные ошибки (5xx, таймаут, сетевой сбой) повторяются, 404 — нет.
- **Выгрузка в CSV** после завершения обхода.
- **Бенчмарк**: сравнение последовательной и параллельной обработки.

---

## Требования

- **.NET SDK 10** (проверить: `dotnet --version`).
- Доступ в интернет (краулер ходит на реальные сайты).
- Один NuGet-пакет — **`System.Threading.Tasks.Dataflow`** (официальная библиотека
  Microsoft, часть платформы .NET). Подключён в `.csproj`, восстановится при сборке.

## Как собрать и запустить

Из корневой папки проекта (где лежит `DistributedWebCrawler.csproj`):

```bash
dotnet build
```

Система — **один** исполняемый файл, запускаемый в разных **ролях**. Для
распределённой работы откройте несколько терминалов.

### Шаг 1. Мастер (терминал №1)

```bash
dotnet run -- master --seed https://books.toscrape.com/ --depth 2 --pages 50
```

Мастер слушает порт `5000` и ждёт подключения воркеров.

### Шаг 2. Воркеры (терминалы №2, №3, ...)

```bash
dotnet run -- worker --master localhost:5000 --id worker-1
dotnet run -- worker --master localhost:5000 --id worker-2
```

Когда воркеры подключатся, мастер начнёт раздавать задачи. В терминале мастера
каждую секунду печатается строка мониторинга. По завершении обхода мастер выводит
итоги, выгружает данные в CSV и предлагает **поиск** по собранным страницам:

```
поиск> history
  1. [7] A Light in the Attic
     https://books.toscrape.com/catalogue/a-light-in-the-attic_1000/index.html
```

### Шаг 3. Проверка отказоустойчивости (по желанию)

Во время обхода закройте окно одного из воркеров (или `Ctrl+C` в нём). Через
несколько секунд мастер заметит пропажу «пульса», напишет об этом и вернёт
незавершённые задачи в очередь — их подхватят оставшиеся воркеры.

### Бенчмарк

```bash
dotnet run -- benchmark --seed https://books.toscrape.com/ --pages 30 --parallelism 8
```

В конце печатается таблица с ускорением (speedup) и эффективностью — эти числа
нужно вписать в `PerformanceReport.md`.

## Как это работает (пошагово)

1. **Запуск.** `Program.cs` по первому аргументу (`master`/`worker`/`benchmark`)
   запускает нужную роль.
2. **Мастер** (`MasterServer`) открывает TCP-порт и кладёт стартовый URL в очередь
   (`BlockingCollection`). Ждёт подключения воркеров.
3. **Воркер** (`WorkerClient`) подключается, регистрируется (`Register`) и раз в
   2 секунды шлёт «пульс» (`Heartbeat`).
4. **Раздача.** Мастер берёт URL из очереди и по выбранной **стратегии**
   (`IDistributionStrategy`) отдаёт его воркеру (`Assign`), запоминая как «в работе».
5. **Обработка.** Воркер кладёт URL в **конвейер** (`CrawlPipeline` на TPL Dataflow):
   загрузка → разбор → выдача. Загрузка делается с **ретраями** при временных ошибках.
   Результат каждой страницы отправляется мастеру (`Result`).
6. **Сбор.** Мастер добавляет данные в **индекс** (`InvertedIndex`) и ставит новые
   ссылки в очередь (если глубина не превышена и домен тот же). Повторные результаты
   по уже обработанному URL игнорируются (защита от дублей).
7. **Мониторинг.** `SystemMonitor` раз в секунду печатает: воркеров, очередь,
   обработано, ошибок, ссылок и скорость.
8. **Отказоустойчивость.** Если от воркера долго нет «пульса», мастер считает его
   упавшим и **возвращает его незавершённые задачи в очередь**.
9. **Завершение и поиск.** Когда незавершённых задач не осталось, мастер печатает
   итоги (в т.ч. число 404 и до 10 примеров таких ссылок) и запускает интерактивный
   поиск по индексу (PLINQ).
10. **Выгрузка в CSV.** Успешно обработанные страницы выгружаются в CSV. Имя файла
    по умолчанию формируется автоматически из хоста и даты-времени, например
    `books.toscrape.com_20260619-230145.csv` (можно задать через `--output`).
    Столбцы: `Url, Depth, Title, Words` (первые 100 слов), `ByteCount, Success`.

| Технология             | Где в коде                                                   |
| ---------------------- | ------------------------------------------------------------ |
| `async/await`          | `PageDownloader`, сетевой обмен, циклы мастера и воркера     |
| TPL / `Task`           | фоновые циклы мастера, запуск задач                          |
| **TPL Dataflow**       | `CrawlPipeline` (конвейер загрузка → разбор → выдача)        |

| `ConcurrentDictionary` | индекс, посещённые/обработанные URL, список воркеров         |
| `BlockingCollection`   | очередь задач (frontier)                                     |
| `ConcurrentQueue`      | накопление результатов для CSV, примеры 404                  |

| TCP-сокеты             | `MasterServer` (`TcpListener`), `WorkerClient` (`TcpClient`) |

| Ретраи                 | `RetryPolicy` + цикл повторов в `PageDownloader`             |
| Паттерн Strategy       | `IDistributionStrategy` + две реализации                     |
| `CancellationToken`    | отмена по `Ctrl+C` и команде `Stop` везде                    |
| `Stopwatch`            | `SystemMonitor`, `BenchmarkRunner`                           |

## Как выполнены запреты задания

- **Без ручного управления потоками** (`Thread`/`Monitor`): используем `Task`,
  Dataflow и потокобезопасные коллекции. `lock` встречается один раз — только для
  аккуратного вывода в консоль, а не для координации задач. Для сериализации записи
  в сетевой поток используется `SemaphoreSlim` (современный async-примитив).
- **Без сторонних библиотек**: единственная зависимость —
  `System.Threading.Tasks.Dataflow` (официальная библиотека Microsoft, прямо
  рекомендованная заданием). HTML разбираем штатными регулярными выражениями.
- **Документация архитектуры** — `SystemArchitecture.md`, `ImplementationGuide.md`,
  `PerformanceReport.md`.
- **Обработка ошибок и отказов** — `try/catch` во всех сетевых операциях, ретраи,
  heartbeat и переназначение задач.
- **Сравнение с последовательной обработкой** — режим `benchmark`.
- **Без глобальных переменных** — состояние хранится в полях объектов и передаётся
  через конструкторы.
- **Мониторинг** — `SystemMonitor` в реальном времени.

## Важные оговорки (честно для защиты)

- HTML разбирается **регулярными выражениями**, потому что специализированные
  библиотеки запрещены. Для учебного примера достаточно, но для «боевого» краулера
  так делать не рекомендуется.
- Краулер **вежливый**: задержка между выдачами задач, ограничение по домену,
  ASCII-заголовок `User-Agent`. Значение `User-Agent` должно быть только из
  ASCII-символов (кириллица в HTTP-заголовках недопустима).
- На реальной сети абсолютное время «прыгает» от запуска к запуску — в бенчмарке
  важно соотношение (ускорение), а не абсолютные секунды.
- Скорость распределённого обхода ограничена параметром `--delay` (пауза в цикле
  раздачи действует глобально). Для большей скорости уменьшайте `--delay`.
