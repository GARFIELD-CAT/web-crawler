# Отчёт о производительности

> **Важно.** Числа в разделах «Пример» приведены для ориентира и помечены как
> _иллюстративные_. Перед сдачей запустите бенчмарк на своей машине и впишите
> **свои** результаты. Команда:
>
> ```bash
> dotnet run -- benchmark --seed https://books.toscrape.com/ --pages 30 --parallelism 8
> ```

---

## 1. Методика измерений

Сравниваются два способа обработки **одного и того же** набора страниц:

1. **Последовательно** — страницы обрабатываются по одной (`await` на каждой).
2. **Параллельно** — страницы проходят через конвейер TPL Dataflow с заданной
   степенью параллелизма.

Время измеряется `Stopwatch`. Измерения идут на реальной сети, поэтому абсолютное
время меняется от запуска к запуску; ключевой показатель — **соотношение** времён.

**Показатели:**

- **Ускорение (speedup):** `S = T_посл / T_парал`
- **Эффективность:** `E = S / P`, где `P` — степень параллелизма.

Рекомендуется выполнить **3–5 прогонов** и взять среднее.

> Примечание про ретраи: в бенчмарке для чистоты замеров повторы лучше отключить
> (`--retries 0`), иначе единичные временные ошибки добавят пауз. На стабильном
> песочнице-сайте это обычно несущественно.

**Условия теста (заполнить своими):**

| Параметр                               | Значение                    |
| -------------------------------------- | --------------------------- |
| Процессор / число ядер                 | **\_\_\_**                  |
| ОЗУ                                    | **\_\_\_**                  |
| ОС                                     | **\_\_\_**                  |
| Версия .NET (`dotnet --version`)       | **\_\_\_**                  |
| Сайт (`--seed`)                        | https://books.toscrape.com/ |
| Число страниц (`--pages`)              | 30                          |
| Степень параллелизма (`--parallelism`) | 8                           |

## 3. Масштабируемость по степени параллелизма

```bash
dotnet run -- benchmark --pages 30 --parallelism 1
dotnet run -- benchmark --pages 30 --parallelism 2
dotnet run -- benchmark --pages 30 --parallelism 4
dotnet run -- benchmark --pages 30 --parallelism 8
dotnet run -- benchmark --pages 30 --parallelism 16
```

| Параллелизм | Время, с | Ускорение | Эффективность |
| ----------: | -------: | --------: | ------------: |
|           1 |   **\_** |      1.00 |         100 % |
|           2 |   **\_** |    **\_** |        **\_** |
|           4 |   **\_** |    **\_** |        **\_** |
|           8 |   **\_** |    **\_** |        **\_** |
|          16 |   **\_** |    **\_** |        **\_** |

### Пример (иллюстративные числа)

| Параллелизм | Время, с | Ускорение | Эффективность |
| ----------: | -------: | --------: | ------------: |
|           1 |     18.5 |       1.0 |         100 % |
|           2 |      9.6 |       1.9 |          96 % |
|           4 |      5.1 |       3.6 |          91 % |
|           8 |      2.9 |       6.4 |          80 % |

Это согласуется с **законом Амдала**: часть работы (сетевые ограничения сайта,
разбор HTML, синхронизация) плохо распараллеливается.

## 4. Масштабируемость по числу воркеров (распределённый режим)

Запускайте мастер и постепенно добавляйте воркеры, фиксируя время полного обхода
(из строки «ИТОГИ ОБХОДА» в логе мастера):

| Число воркеров | Время обхода, с | Ускорение |
| -------------: | --------------: | --------: |
|              1 |          **\_** |      1.00 |
|              2 |          **\_** |    **\_** |
|              3 |          **\_** |    **\_** |

> **Важно.** В распределённом режиме суммарную скорость ограничивает параметр
> `--delay`: пауза в цикле раздачи действует **глобально**, поэтому мастер выдаёт
> не более `1000/--delay` задач в секунду независимо от числа воркеров. При оценке
> масштабируемости держите `--delay` одинаковым; чтобы воркеры не простаивали,
> уменьшайте `--delay`. Грубая оценка времени обхода: `число_страниц × --delay`.

---

## 5. Почему параллельная версия быстрее

Веб-краулер — задача с преобладанием ввода-вывода (I/O-bound): бо́льшую часть времени
программа ждёт ответа сервера. В последовательном режиме ожидания идут одно за другим;
в параллельном — **накладываются друг на друга**, и суммарное время резко сокращается.
Поэтому выигрыш идёт не столько от вычислений, сколько от совмещённого ожидания.

---

## 6. Узкие места

- **Глобальная пауза `--delay`** в цикле раздачи — главный ограничитель скорости
  распределённого обхода (см. раздел 4).
- **Скорость и лимиты целевого сайта.** Медленные ответы или ограничение частоты —
  потолок, не обходимый числом потоков.
- **Разбор HTML регулярными выражениями.** Это CPU-работа, заметная на больших
  страницах.
- **Единственный мастер.** Все результаты сходятся в один узел (он же — единая точка
  отказа).
- **Накладные расходы сети.** Сериализация в JSON и пересылка сообщений добавляют
  небольшую задержку (обычно незначительную на фоне загрузки страниц).
- **Ретраи.** При множестве временных ошибок повторы с паузами увеличивают время;
  для чистых замеров их отключают (`--retries 0`).

---

## 7. Рекомендации по оптимизации

- Подбирать `--parallelism` под число ядер и допустимую нагрузку на сайт.
- Уменьшать `--delay`, если целевой сайт это допускает (ускоряет распределённый режим).
- Для CPU-части (разбор) — специализированный парсер (вне рамок задания).
- Для снижения нагрузки на мастер — агрегировать результаты пачками и/или
  распределять индекс между несколькими узлами.

---

## 8. Выводы

- Параллельная обработка на TPL Dataflow даёт **существенное ускорение** против
  последовательной (по примеру — около ×6 при параллелизме 8).
- Система **масштабируется** и по параллелизму внутри воркера, и по числу воркеров;
  в распределённом режиме важно не забывать про глобальную паузу `--delay`.
- Эффективность ожидаемо снижается с ростом ресурсов (закон Амдала), что определяет
  разумный предел масштабирования.
