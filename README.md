# Распределённый веб-краулер (учебный проект)

Распределённая система для параллельного сбора данных с веб-сайтов на C# и .NET 10.
Проект демонстрирует современные подходы к параллельному и распределённому
программированию: `async/await`, **TPL Dataflow**, **PLINQ**, потокобезопасные
коллекции, атомарные операции `Interlocked`, а также **мастер-рабочую архитектуру**
с сетевым обменом по TCP, механизмом «пульса» (heartbeat) и переназначением задач
при отказе узлов.

> Стартовый сайт по умолчанию — `https://books.toscrape.com` — это учебный
> «песочница»-сайт, специально созданный для практики краулинга. Его можно
> безопасно обходить и менять на любой другой через параметр `--seed`.

---

## Что умеет система

- **Параллельная загрузка** веб-страниц (асинхронные HTTP-запросы).
- **Извлечение данных** из HTML: заголовок, слова, ссылки.
- **Обход ссылок** с ограничением глубины и только в пределах одного домена.
- **Индексация и поиск** по собранным данным (инвертированный индекс + поиск).
- **Распределение задач** между несколькими рабочими узлами (воркерами).
- **Мониторинг** состояния системы в реальном времени и статистика.
- **Отказоустойчивость**: если воркер «падает», его задачи получают другие.
- **Бенчмарк**: сравнение последовательной и параллельной обработки.

---

## Требования

- **.NET SDK 10** (проект собран и протестирован под `10.0.x`).
  Проверить версию: `dotnet --version`.
- Доступ в интернет (краулер ходит на реальные сайты).
- Один NuGet-пакет — **`System.Threading.Tasks.Dataflow`** (официальная библиотека
  Microsoft, часть платформы .NET). Он подключён в `.csproj` и восстановится
  автоматически при первой сборке.

## Как собрать и запустить

Из корневой папки проекта (где лежит `DistributedWebCrawler.csproj`):

```bash
# Восстановить зависимости и собрать
dotnet build
```

Система состоит из **одного** исполняемого файла, который запускается в разных
**ролях**. Для распределённой работы откройте несколько терминалов.

### Шаг 1. Запустить мастер (терминал №1)

```bash
dotnet run -- master --seed https://books.toscrape.com/ --depth 3 --pages 50
```

Мастер начнёт слушать порт `5000` и будет ждать подключения воркеров.

### Шаг 2. Запустить один или несколько воркеров (терминалы №2, №3, ...)

```bash
dotnet run -- worker --master localhost:5000 --id worker-1
```

```bash
dotnet run -- worker --master localhost:5000 --id worker-2
```

Как только воркеры подключатся, мастер начнёт раздавать им задачи. В терминале
мастера каждую секунду будет печататься строка мониторинга. Когда обход завершится,
мастер покажет итоги и предложит **поиск** по собранным страницам:

```
поиск> history
  1. [7] A Light in the Attic
     https://books.toscrape.com/catalogue/a-light-in-the-attic_1000/index.html
  ...
```

### Шаг 3. Проверить отказоустойчивость (по желанию)

Во время обхода **закройте окно одного из воркеров** (или нажмите там `Ctrl+C`).
Мастер через несколько секунд заметит пропажу «пульса», напишет об этом в лог и
вернёт незавершённые задачи в очередь — их подхватят оставшиеся воркеры.

### Бенчмарк (последовательно vs параллельно)

```bash
dotnet run -- benchmark --seed https://books.toscrape.com/ --pages 30 --parallelism 8
```

В конце будет напечатана таблица с ускорением (speedup) и эффективностью —
эти числа нужно вписать в `PerformanceReport.md`.

## Как это работает (пошагово, для новичка)

1. **Запуск.** `Program.cs` смотрит на первый аргумент (`master`, `worker` или
   `benchmark`) и запускает соответствующую роль.

2. **Мастер** (`MasterServer`) открывает TCP-порт и кладёт стартовый URL в очередь
   задач (`BlockingCollection`). Он ждёт, пока подключатся воркеры.

3. **Воркер** (`WorkerClient`) подключается к мастеру, представляется (сообщение
   `Register`) и начинает раз в 2 секунды слать «пульс» (`Heartbeat`).

4. **Раздача задач.** Мастер берёт URL из очереди и, с помощью выбранной
   **стратегии** (`IDistributionStrategy`), решает, какому воркеру его отдать
   (сообщение `Assign`). Отправленная задача запоминается как «в работе».

5. **Обработка.** Воркер кладёт полученный URL в свой **конвейер**
   (`CrawlPipeline` на TPL Dataflow): станция «загрузка» (асинхронно скачивает
   HTML), станция «разбор» (вытаскивает заголовок, слова, ссылки), станция
   «выдача» (отправляет результат мастеру сообщением `Result`).

6. **Сбор результатов.** Мастер получает `Result`, добавляет данные в
   **инвертированный индекс** (`InvertedIndex`) и ставит **новые найденные ссылки**
   в очередь (если глубина не превышена и домен тот же). Так обход идёт «вширь».

7. **Мониторинг.** Параллельно `SystemMonitor` раз в секунду печатает: сколько
   воркеров живо, сколько задач в очереди, сколько страниц обработано и скорость.

8. **Отказоустойчивость.** Если от воркера давно не было «пульса», мастер считает
   его упавшим, удаляет и **возвращает его незавершённые задачи в очередь**.

9. **Завершение.** Когда незавершённых задач не осталось, мастер печатает итоги и
   запускает **интерактивный поиск** по собранным данным (поиск использует PLINQ).

## Где какая технология используется

| Технология             | Где в коде                                                               |
| ---------------------- | ------------------------------------------------------------------------ |
| `async/await`          | `PageDownloader`, весь сетевой обмен, циклы мастера и воркера            |
| TPL / `Task`           | фоновые циклы мастера, запуск задач                                      |
| **TPL Dataflow**       | `CrawlPipeline` (конвейер загрузка → разбор → выдача)                    |
| **PLINQ**              | `InvertedIndex.Search` (`AsParallel`)                                    |
| `ConcurrentDictionary` | индекс, множество посещённых URL, список воркеров                        |
| `BlockingCollection`   | очередь задач (frontier) в мастере                                       |
| `Interlocked`          | счётчики в `Statistics`, `InvertedIndex`, `MasterServer`                 |
| TCP-сокеты             | `MasterServer` (`TcpListener`), `WorkerClient` (`TcpClient`)             |
| Heartbeat              | `WorkerClient.HeartbeatLoopAsync` + `MasterServer.HeartbeatMonitorAsync` |
| Паттерн Strategy       | `IDistributionStrategy` + две реализации                                 |
| `CancellationToken`    | отмена по `Ctrl+C` и команде `Stop` во всех компонентах                  |
| `Stopwatch`            | `SystemMonitor`, `BenchmarkRunner`                                       |

## Как выполнены запреты задания

- **Без ручного управления потоками** (`Thread`/`Monitor`): используем `Task`,
  Dataflow и потокобезопасные коллекции. `lock` встречается один раз — только для
  аккуратного вывода в консоль, а не для координации задач.
- **Без сторонних библиотек**: единственная зависимость —
  `System.Threading.Tasks.Dataflow` (официальная библиотека Microsoft, прямо
  рекомендованная заданием). HTML разбираем штатными регулярными выражениями.
- **Документация архитектуры** — файлы `SystemArchitecture.md`,
  `ImplementationGuide.md`, `PerformanceReport.md`.
- **Обработка ошибок и отказов** — `try/catch` во всех сетевых операциях,
  heartbeat и переназначение задач.
- **Сравнение с последовательной обработкой** — режим `benchmark`.
- **Без глобальных переменных** — состояние хранится в полях объектов и передаётся
  через конструкторы.
- **Мониторинг** — `SystemMonitor` в реальном времени.

## Важные оговорки (честно для защиты)

- HTML разбирается **регулярными выражениями**, потому что специализированные
  библиотеки запрещены. Для учебного примера этого достаточно, но для «боевого»
  краулера так делать не рекомендуется — там используют настоящие HTML-парсеры.
- Краулер **вежливый**: задержка между запросами, ограничение по домену, заголовок
  `User-Agent`. Обходите только те сайты, где это разрешено.
- На реальной сети время выполнения «прыгает» от запуска к запуску — поэтому в
  бенчмарке важно не абсолютное время, а **соотношение** (ускорение).
