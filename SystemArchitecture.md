# Архитектура системы

Документ описывает устройство распределённого веб-краулера: из каких компонентов
он состоит, как они взаимодействуют, почему выбраны те или иные технологии, и как
обеспечиваются масштабируемость и отказоустойчивость.

---

## 1. Общая идея

Система построена по схеме **«мастер — рабочие»** (master–worker):

- **Мастер** — единый координатор. Хранит очередь адресов, раздаёт их рабочим
  узлам, собирает результаты, строит индекс, следит за состоянием узлов.
- **Воркеры** — рабочие узлы. Получают адреса, скачивают и разбирают страницы,
  возвращают результаты мастеру.

Мастер и воркеры — это **отдельные процессы**, которые обычно запускаются в разных
терминалах (а в реальности — на разных машинах) и общаются по сети через **TCP**.

```
                     ┌──────────────────────────────────────┐
                     │                МАСТЕР                  │
                     │                                        │
   стартовый URL ──► │  Очередь задач (BlockingCollection)    │
                     │  Множество посещённых (ConcurrentDict) │
                     │  Стратегия распределения (Strategy)    │
                     │  Инвертированный индекс (ConcurrentDict)│
                     │  Монитор + статистика (Interlocked)    │
                     │  Контроль heartbeat                     │
                     └───────┬───────────────┬───────────────┘
                             │   TCP-сокеты   │
              ┌──────────────┘                └──────────────┐
              ▼                                               ▼
   ┌────────────────────┐                        ┌────────────────────┐
   │      ВОРКЕР 1       │                        │      ВОРКЕР N       │
   │  Конвейер Dataflow: │                        │  Конвейер Dataflow: │
   │  загрузка → разбор  │         ...            │  загрузка → разбор  │
   │  → отправка         │                        │  → отправка         │
   │  + heartbeat        │                        │  + heartbeat        │
   └─────────┬──────────┘                        └─────────┬──────────┘
             │ HTTP-запросы                                 │ HTTP-запросы
             ▼                                               ▼
        ┌─────────────────────── Интернет (веб-сайт) ───────────────────────┐
```

## 2. Компоненты и зоны ответственности

Проект разбит на слои; каждый слой решает свою задачу (принцип разделения
ответственности).

| Слой | Класс(ы) | Ответственность |
|------|----------|-----------------|
| Данные | `CrawlTask`, `PageData`, `Message` | структуры, которыми обмениваются компоненты |
| Контракты | `ILogger`, `IHtmlParser`, `IDistributionStrategy` | интерфейсы для подмены реализаций |
| Обход | `PageDownloader`, `HtmlParser`, `CrawlPipeline` | скачивание и разбор страниц |
| Индексация | `InvertedIndex` | хранение данных и поиск |
| Распределение | `RoundRobinStrategy`, `LeastLoadedStrategy` | выбор воркера для задачи |
| Мониторинг | `Statistics`, `SystemMonitor` | счётчики и вывод состояния |
| Сеть | `MessageProtocol`, `WorkerInfo`, `MasterServer`, `WorkerClient` | TCP-обмен и координация |
| Бенчмарк | `BenchmarkRunner` | сравнение производительности |
| Запуск | `Program` | разбор аргументов и выбор роли |

## 3. Поток данных при обходе

```
[Очередь URL] ──Assign──► [Воркер] ──► [Загрузка HTML] ──► [Разбор HTML]
      ▲                                                          │
      │                                                          ▼
      └──────────── новые ссылки ◄──Result── [Мастер] ◄──── [PageData]
                                                  │
                                                  ▼
                                          [Инвертированный индекс] ──► [Поиск]
```

1. Мастер достаёт URL из очереди и отправляет воркеру (`Assign`).
2. Воркер скачивает страницу, разбирает её и возвращает `PageData` (`Result`).
3. Мастер добавляет данные в индекс, а найденные ссылки — обратно в очередь.
4. Цикл повторяется, пока есть задачи и не превышены лимиты глубины/числа страниц.

---

## 4. Протокол сетевого обмена

Так как TCP — это непрерывный поток байт без границ, используется **кадрирование
длиной**: перед каждым сообщением передаётся 4 байта с его размером.

```
[ 4 байта: длина N ][ N байт: JSON-сообщение ]
```

Все сообщения — это один класс `Message` с полем `Type`:

| Тип | Направление | Назначение |
|-----|-------------|------------|
| `Register` | воркер → мастер | регистрация воркера |
| `Heartbeat` | воркер → мастер | «я жив» (раз в 2 с) |
| `Result` | воркер → мастер | результат обработки страницы |
| `Assign` | мастер → воркер | выдача задачи |
| `Stop` | мастер → воркер | команда остановки |

Сериализация — `System.Text.Json` (встроенная в .NET).

## 5. Обоснование выбора технологий

- **`async/await` + `HttpClient`.** Веб-краулер почти всё время ждёт ответа сети.
  Асинхронность позволяет во время ожидания не держать поток занятым, а заниматься
  другими страницами. Это даёт основной прирост производительности.

- **TPL Dataflow.** Обработка страницы естественно делится на этапы
  (скачать → разобрать → выдать). Dataflow позволяет описать это как конвейер из
  «блоков», где каждый блок параллелит свою работу, а библиотека сама управляет
  потоками и передачей данных между этапами. Это проще и безопаснее, чем вручную
  создавать потоки.

- **PLINQ.** Поиск и ранжирование результатов — это операции над коллекцией.
  PLINQ (`AsParallel`) выполняет такой запрос на нескольких ядрах одной строкой кода.

- **Потокобезопасные коллекции.** `ConcurrentDictionary` (индекс, посещённые URL,
  список воркеров) и `BlockingCollection` (очередь задач) безопасно работают при
  одновременном доступе из многих потоков — без ручных блокировок.

- **`Interlocked`.** Счётчики (обработано страниц, скачано байт и т.п.) обновляются
  атомарно, без гонок и без накладных расходов на блокировки.

- **TCP-сокеты.** Базовый, входящий в .NET способ сетевого взаимодействия между
  процессами/машинами; не требует внешних библиотек.

- **Паттерн Strategy.** Алгоритм распределения задач вынесен в интерфейс, поэтому
  его можно менять (round-robin / least-loaded), не трогая мастер.

## 6. Масштабируемость

- **Горизонтальное масштабирование.** Чтобы ускорить обход, достаточно запустить
  больше воркеров — мастер автоматически начнёт раздавать им задачи. Воркеры можно
  добавлять прямо во время работы (динамическое масштабирование).
- **Балансировка нагрузки.** Стратегия `Least-Loaded` отдаёт задачу наименее
  загруженному воркеру, что выравнивает нагрузку при разной скорости узлов.
- **Параллелизм внутри воркера.** Каждый воркер обрабатывает несколько страниц
  одновременно (параметр `--parallelism`), используя конвейер Dataflow.
- **Минимум накладных расходов.** Координация построена на неблокирующих
  потокобезопасных коллекциях и атомарных счётчиках, а не на «тяжёлых» блокировках.

---

## 7. Отказоустойчивость

- **Heartbeat.** Воркер раз в 2 секунды шлёт «пульс». Мастер хранит время
  последнего пульса каждого воркера.
- **Обнаружение отказа.** Если пульса нет дольше 6 секунд, мастер считает воркер
  упавшим и удаляет его из списка активных.
- **Переназначение задач.** Все задачи, которые упавший воркер не успел завершить,
  возвращаются в общую очередь, и их подхватывают оставшиеся воркеры.
- **Изоляция ошибок.** Ошибка на одной странице (например, недоступный URL) не
  ломает систему: она фиксируется как неуспех, и обход продолжается.
- **Корректная остановка.** По `Ctrl+C` или команде `Stop` все компоненты
  завершаются через общий `CancellationToken`, не теряя уже собранные данные.

## 8. Возможные улучшения (вне рамок учебного задания)

- Настоящий HTML-парсер вместо регулярных выражений.
- Учёт `robots.txt` и отдельная задержка на каждый домен.
- Сохранение индекса на диск и восстановление после перезапуска.
- Резервный мастер (сейчас мастер — единственная точка отказа).
- Сжатие сетевых сообщений для экономии трафика.
