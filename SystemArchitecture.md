# Архитектура системы

Документ описывает устройство распределенного веб-краулера: компоненты, их
взаимодействие, обоснование технологий, масштабируемость и отказоустойчивость.

---

## 1. Общая идея

Система построена по схеме **«мастер — рабочие»** (master–worker):

- **Мастер** — единый координатор. Хранит очередь адресов, раздает их рабочим
  узлам, собирает результаты, строит индекс, следит за состоянием узлов и в конце
  выгружает данные в CSV.
- **Воркеры** — рабочие узлы. Получают адреса, скачивают (с ретраями) и разбирают
  страницы, возвращают результаты мастеру.

Мастер и воркеры — **отдельные процессы** (обычно в разных терминалах, а в реальности
на разных машинах), общающиеся по **TCP**.

```
                     ┌──────────────────────────────────────┐
                     │                МАСТЕР                  │
   стартовый URL ──► │  Очередь задач (BlockingCollection)    │
                     │  Посещенные / обработанные (ConcurrentD)│
                     │  Стратегия распределения (Strategy)    │
                     │  Инвертированный индекс (ConcurrentD)  │
                     │  Монитор + статистика (Interlocked)    │
                     │  Контроль heartbeat + переназначение   │
                     │  Выгрузка в CSV                         │
                     └───────┬───────────────┬───────────────┘
                             │   TCP-сокеты   │
              ┌──────────────┘                └──────────────┐
              ▼                                               ▼
   ┌────────────────────┐                        ┌────────────────────┐
   │      ВОРКЕР 1       │                        │      ВОРКЕР N       │
   │  Конвейер Dataflow: │                        │  Конвейер Dataflow: │
   │  загрузка(+ретраи)  │         ...            │  загрузка(+ретраи)  │
   │  → разбор → выдача  │                        │  → разбор → выдача  │
   │  + heartbeat        │                        │  + heartbeat        │
   └─────────┬──────────┘                        └─────────┬──────────┘
             │ HTTP-запросы                                 │ HTTP-запросы
             ▼                                               ▼
        ┌─────────────────────── Интернет (веб-сайт) ───────────────────────┐
```

## 2. Компоненты и зоны ответственности

| Слой          | Класс(ы)                                                        | Ответственность                   |
| ------------- | --------------------------------------------------------------- | --------------------------------- |
| Данные        | `CrawlTask`, `PageData`, `Message`                              | структуры обмена                  |
| Контракты     | `ILogger`, `IHtmlParser`, `IDistributionStrategy`               | интерфейсы для подмены реализаций |
| Обход         | `PageDownloader`, `RetryPolicy`, `HtmlParser`, `CrawlPipeline`  | загрузка (с ретраями) и разбор    |
| Индексация    | `InvertedIndex`                                                 | хранение данных и поиск           |
| Распределение | `RoundRobinStrategy`, `LeastLoadedStrategy`                     | выбор воркера для задачи          |
| Мониторинг    | `Statistics`, `SystemMonitor`                                   | счетчики и вывод состояния        |
| Сеть          | `MessageProtocol`, `WorkerInfo`, `MasterServer`, `WorkerClient` | TCP-обмен и координация           |
| Бенчмарк      | `BenchmarkRunner`                                               | сравнение производительности      |
| Выгрузка      | `CsvExporter`                                                   | запись результатов в CSV          |
| Запуск        | `Program`                                                       | разбор аргументов и выбор роли    |

## 3. Поток данных при обходе

```
[Очередь URL] ──Assign──► [Воркер] ──► [Загрузка(+ретраи)] ──► [Разбор HTML]
      ▲                                                              │
      │                                                              ▼
      └──────────── новые ссылки ◄──Result── [Мастер] ◄──────── [PageData]
                                                  │
                                  ┌───────────────┼───────────────┐
                                  ▼               ▼               ▼
                          [Индекс/Поиск]   [Защита от дублей]  [Накопление для CSV]
```

1. Мастер достает URL из очереди и отправляет воркеру (`Assign`).
2. Воркер скачивает (повторяя временные ошибки), разбирает и возвращает `PageData`.
3. Мастер проверяет, не обработан ли URL уже (дедупликация), добавляет данные в
   индекс, копит успешные результаты для CSV, а найденные ссылки — обратно в очередь.
4. Цикл повторяется, пока есть задачи и не превышены лимиты глубины/числа страниц.

## 4. Протокол сетевого обмена

TCP — это поток байт без границ сообщений, поэтому применяется **кадрирование
длиной**: перед каждым сообщением передается 4 байта с его размером.

```
[ 4 байта: длина N ][ N байт: JSON-сообщение ]
```

Все сообщения — один класс `Message` с полем `Type`:

| Тип         | Направление     | Назначение                   |
| ----------- | --------------- | ---------------------------- |
| `Register`  | воркер → мастер | регистрация воркера          |
| `Heartbeat` | воркер → мастер | «я жив» (раз в 2 с)          |
| `Result`    | воркер → мастер | результат обработки страницы |
| `Assign`    | мастер → воркер | выдача задачи                |
| `Stop`      | мастер → воркер | команда остановки            |

Сериализация — `System.Text.Json`. Запись в один сетевой поток сериализуется
семафором (`SemaphoreSlim`), чтобы байты разных сообщений не перемешались.

## 5. Обоснование выбора технологий

- **`async/await` + `HttpClient`.** Краулер почти все время ждет ответа сети.
  Асинхронность позволяет не держать поток занятым во время ожидания — отсюда
  основной прирост производительности.
- **TPL Dataflow.** Обработка страницы делится на этапы (скачать → разобрать →
  выдать). Dataflow описывает это как конвейер из «блоков»; каждый блок параллелит
  свою работу, библиотека сама управляет потоками. Это проще и безопаснее ручных
  `Thread`.
- **PLINQ.** Ранжирование результатов поиска — операция над коллекцией; `AsParallel`
  распараллеливает ее одной строкой.
- **Потокобезопасные коллекции.** `ConcurrentDictionary` (индекс, посещенные/
  обработанные URL, воркеры) и `BlockingCollection` (очередь) безопасны при
  одновременном доступе без ручных блокировок.
- **`Interlocked`.** Счетчики обновляются атомарно, без гонок и тяжелых блокировок.
- **TCP-сокеты.** Базовый, входящий в .NET способ межпроцессного обмена.
- **Паттерн Strategy.** Алгоритм распределения вынесен в интерфейс — его можно
  менять, не трогая мастер.
- **Ретраи (`RetryPolicy`).** Временные сбои (5xx, таймаут, сеть) повторяются с
  экспоненциальной паузой; постоянные (404) — нет. Набор повторяемых статусов
  настраивается.

## 6. Масштабируемость

- **Горизонтальное масштабирование.** Чтобы ускорить обход, запускают больше
  воркеров — мастер сам начнет раздавать им задачи. Воркеры можно добавлять во время
  работы (динамическое масштабирование).
- **Балансировка нагрузки.** Стратегия `Least-Loaded` отдает задачу наименее
  загруженному воркеру.
- **Параллелизм внутри воркера.** Каждый воркер обрабатывает несколько страниц
  одновременно (`--parallelism`) через конвейер Dataflow.
- **Регулировка нагрузки на сайт.** Пауза `--delay` в цикле раздачи ограничивает
  частоту запросов (вежливость); это же ограничивает суммарную скорость обхода.

## 7. Отказоустойчивость

Различаются два уровня сбоев:

- **Временная ошибка страницы** (5xx, таймаут, сетевой сбой) — лечится **ретраями**
  внутри воркера (`RetryPolicy` + `PageDownloader`): запрос повторяется несколько раз
  с нарастающей паузой. Ошибка 404 не повторяется.
- **Падение воркера целиком** — лечится **переназначением задач**:
  - воркер раз в 2 с шлет «пульс»; мастер хранит время последнего пульса;
  - если пульса нет дольше 6 секунд, воркер считается упавшим и удаляется;
  - все его незавершенные задачи возвращаются в очередь и достаются другим воркерам.

Дополнительно:

- **Защита от дублей.** Если задача «упавшего» (а на деле живого) воркера была
  переназначена, результат может прийти дважды. Множество обработанных URL
  (`_completed`) гарантирует, что каждая страница учитывается, индексируется и
  выгружается ровно один раз.
- **Изоляция ошибок.** Ошибка на одной странице не ломает систему — фиксируется и
  обход продолжается.
- **Корректная остановка.** По `Ctrl+C`/`Stop` все компоненты завершаются через общий
  `CancellationToken`, не теряя собранные данные; CSV выгружается в конце.

## 8. Возможные улучшения (вне рамок задания)

- Настоящий HTML-парсер вместо регулярных выражений.
- Учет `robots.txt`, фильтры путей и отдельная задержка на каждый домен.
- Учет заголовка `Retry-After` при 429/503.
- Сохранение индекса/очереди на диск и восстановление после перезапуска.
- Резервный мастер (сейчас мастер — единственная точка отказа).
