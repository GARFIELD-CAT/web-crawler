# Отчёт о производительности

> **Важно.** Числа в разделах «Пример» приведены для ориентира и помечены как
> _иллюстративные_. Перед сдачей запустите бенчмарк на своей машине и впишите
> **свои** результаты в таблицы ниже. Команда:
>
> ```bash
> dotnet run -- benchmark --seed https://books.toscrape.com/ --pages 30 --parallelism 8
> ```

---

## 1. Методика измерений

Сравниваются два способа обработки **одного и того же** набора страниц:

1. **Последовательно** — страницы обрабатываются по одной (`await` на каждой).
2. **Параллельно** — страницы проходят через конвейер TPL Dataflow с заданной
   степенью параллелизма.

Замер времени выполняется секундомером `Stopwatch`. Поскольку измерения идут на
реальной сети, абсолютное время меняется от запуска к запуску, поэтому ключевой
показатель — **соотношение** времён, а не их абсолютные значения.

**Показатели:**

- **Ускорение (speedup):** `S = T_посл / T_парал`
- **Эффективность:** `E = S / P`, где `P` — степень параллелизма.

Рекомендуется выполнить **3–5 прогонов** каждого режима и взять среднее, чтобы
сгладить случайные колебания сети.

**Условия теста (заполнить своими):**

| Параметр                               | Значение                        |
| -------------------------------------- | ------------------------------- |
| Процессор / число ядер                 | **\_\_\_**                      |
| ОЗУ                                    | **\_\_\_**                      |
| ОС                                     | **\_\_\_**                      |
| Версия .NET                            | `dotnet --version` → **\_\_\_** |
| Сайт (`--seed`)                        | https://books.toscrape.com/     |
| Число страниц (`--pages`)              | 100                             |
| Степень параллелизма (`--parallelism`) | 10                              |

## 3. Масштабируемость по степени параллелизма

Запустите бенчмарк с разными значениями `--parallelism` и заполните таблицу:

```bash
dotnet run -- benchmark --pages 30 --parallelism 1
dotnet run -- benchmark --pages 30 --parallelism 2
dotnet run -- benchmark --pages 30 --parallelism 4
dotnet run -- benchmark --pages 30 --parallelism 8
dotnet run -- benchmark --pages 30 --parallelism 16
```

|           2 |   **\_** |    **\_** |        **\_** |
|           4 |   **\_** |    **\_** |        **\_** |
|           8 |   **\_** |    **\_** |        **\_** |
|          16 |   **\_** |    **\_** |        **\_** |

### Пример (иллюстративные числа)

| Параллелизм | Время, с | Ускорение | Эффективность |
| ----------: | -------: | --------: | ------------: |
|           1 |     18.5 |       1.0 |         100 % |

|           8 |      2.9 |       6.4 |          80 % |

**Наблюдение (типичное):** с ростом параллелизма ускорение растёт, но эффективность

сайта, разбор HTML, синхронизация) плохо распараллеливается, поэтому после
некоторого порога добавление потоков почти не помогает.

## 4. Масштабируемость по числу воркеров (распределённый режим)

Запустите мастер и постепенно добавляйте воркеры, фиксируя время полного обхода
(из строки «ИТОГИ ОБХОДА» в логе мастера):

| Число воркеров | Время обхода, с | Ускорение |
| -------------: | --------------: | --------: |
|              1 |          **\_** |      1.00 |
|              2 |          **\_** |    **\_** |
|              3 |          **\_** |    **\_** |

> Примечание: на распределённый режим влияет параметр `--delay` (пауза вежливости
> между выдачами задач). Для оценки масштабируемости держите его одинаковым во всех
> прогонах.

## 5. Почему параллельная версия быстрее

Веб-краулер — это **задача с преобладанием ввода-вывода** (I/O-bound): бо́льшую
часть времени программа просто ждёт ответа от сервера. В последовательном режиме во
время ожидания одной страницы ничего больше не происходит. В параллельном режиме
ожидания разных страниц **накладываются друг на друга**, и суммарное время резко
сокращается. Именно поэтому здесь ускорение может превышать «интуитивный» предел —
выигрыш идёт не столько от вычислений, сколько от совмещённого ожидания.

---

## 6. Узкие места

- **Скорость и лимиты целевого сайта.** Сервер может отвечать медленно или
  ограничивать частоту запросов — это потолок, который не обойти увеличением потоков.
- **Разбор HTML регулярными выражениями.** Это вычислительная (CPU) работа; на
  больших страницах она заметна. Настоящий парсер мог бы быть эффективнее.
- **Единственный мастер.** Все результаты сходятся в один узел; при очень большом
  числе воркеров он может стать узким местом (и является единой точкой отказа).
- **Накладные расходы сети.** Сериализация в JSON и пересылка сообщений добавляют
  небольшую задержку (обычно незначительную на фоне времени загрузки страниц).
- **Пауза вежливости (`--delay`).** Намеренно ограничивает скорость в
  распределённом режиме ради корректного отношения к сайту.

---

## 7. Рекомендации по оптимизации

- Подбирать `--parallelism` под число ядер и допустимую нагрузку на сайт
  (по таблице из раздела 3 видно точку, после которой рост почти прекращается).
- Уменьшать `--delay`, только если целевой сайт это допускает.
- Для CPU-части (разбор) — заменить регулярные выражения на специализированный
  парсер (вне рамок задания из-за запрета внешних библиотек).
- Для снижения нагрузки на мастер — агрегировать результаты пачками и/или
  распределять индекс между несколькими узлами.

---

## 8. Выводы

- Параллельная обработка на TPL Dataflow даёт **существенное ускорение** по
  сравнению с последовательной (по примеру — около ×6 при параллелизме 8).
- Система **масштабируется**: ускорение растёт и при увеличении параллелизма
  внутри воркера, и при добавлении новых воркеров.
- Эффективность ожидаемо снижается с ростом ресурсов (закон Амдала), что и
  определяет разумный предел масштабирования.
